# Spec To Implementation

Convert a local Markdown spec into linked implementation plans, tasks, and status-ready downstream outputs.

## Quick start
1) Locate and read the local spec or requirements artifact.
2) Parse requirements, constraints, ambiguities, and dependencies.
3) Choose the planning depth: quick implementation plan or multi-phase plan.
4) Produce local Markdown plan and task outputs with clear traceability back to the source spec.
5) Link the outputs and record blockers, risks, and status updates as work progresses.

## Workflow

### 1) Locate and read the spec
- Read the local Markdown spec or requirements artifact first.
- Scan for requirements, acceptance criteria, constraints, priorities, and known gaps.
- Capture ambiguities or clarifications before drafting a plan.

### 2) Choose plan depth
- Simple change → concise implementation plan.
- Multi-phase feature, migration, or cross-team work → standard implementation plan with phases and dependencies.

### 3) Create plan outputs
- Produce local Markdown outputs that include:
  - overview
  - linked source spec
  - requirements summary
  - phases or workstreams
  - dependencies and risks
  - success criteria

### 4) Create task-ready breakdown
- Break work into concrete downstream tasks.
- Keep tasks traceable to the source spec and the plan.
- Include context, objective, acceptance criteria, dependencies, and resources when relevant.

### 5) Track progress
- Keep linked outputs consistent.
- Record blockers, decisions, and milestone summaries in local Markdown rather than relying on external systems by default.

## Notes

- This is a local-Markdown adaptation of a Notion-based spec-to-implementation pattern.
- It is a downstream bridge skill, not a core Business Analyst discovery skill.
- Use it after Business Analyst outputs are mature enough to support implementation planning.
