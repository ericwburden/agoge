# Orpheum 1.1.0 Release Notes

## Summary

Orpheum 1.1.0 adds a first-class Definition of Done to the planning stack so implementation-phase completion rules are explicit, durable, and shared across downstream roles instead of being inferred slice by slice.

## Highlights

- Added a new `definition-of-done` artifact and `definition-of-done.check.md` check to the Technical Planner artifact chain.
- `project-planning` now produces a reviewed project-level Definition of Done alongside implementation strategy and sequencing artifacts.
- Technical Planner guidance now makes the boundary explicit between enduring Definition of Done rules and slice-local exit criteria.
- Implementation artifacts now preserve Definition-of-Done alignment and status so downstream code review and verification consumers can see whether a slice is merely implemented or is actually aligned with the project’s completion standard.

## Upgrade Notes

- Teams using `project-planning` should now expect Technical Planner outputs to include a Definition of Done artifact before implementation begins.
- Existing implementation planning artifacts should keep slice exit criteria as slice-local tailoring and move enduring completion rules into the new Definition of Done artifact.
- Downstream implementation package consumers should treat Definition-of-Done status as a project-level completion signal, not as a replacement for independent review, verification, or release readiness.
